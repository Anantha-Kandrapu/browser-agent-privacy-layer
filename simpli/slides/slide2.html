Architecture — Proxy Flow

What the Proxy Does on Each Leg

Request — runtime → model

  1. Redact screenshot (detector), store spans + real values in the vault
  2. Scrub history: vault match, then semantic reclassify (Rampart) as backstop
  3. Inject placeholder instructions (e.g. EMAIL_1_a3f9) into the request

Response — model → runtime

  1. Parse the returned action
  2. Resolve placeholders in executed fields only, never in reasoning text
  3. Gated classes (e.g. API keys) are consulted with the mediator before resolving
  4. Fail closed: no match → no forward, ever
Per-class safety levels govern resolution: hide‑use (freely resolvable), approval (needs mediator sign-off), blocked (never stored, never resolvable). Nothing is persisted to disk, and logs never carry real values or frames.
